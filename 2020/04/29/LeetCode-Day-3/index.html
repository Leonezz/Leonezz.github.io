$algorithms$ $Medium (54.30\%)$

思路

上午打开这一题，一直做了俩小时才做出来。看起来似乎没那么难，根昨天做的[102]题有点相似。本来想着套用一下[102]题的模板，增加一点点东西解出来，没想到一下卡了半天。
打开讨论区发现一个思路很好，照着写出了题解。原来只要把[102]题的答案稍微变一下就可以很轻松做出来：在使用队列层序遍历二叉树的时候，每次直接读取整层的节点，同时把下层的节点放进队列。这样队列进入循环时，里面存的一定是一整层的节点，这样每层的节点数就有了。直接初始化一个节点数大小的向量，然后把这层的节点按照下标规律放进向量即可。

（这是一个 交互式问题 ）

给你一个 山脉数组 mountainArr，请你返回能够使得 mountainArr.get(index) 等于 target 最小 的下标 index 值。

如果不存在这样的下标 index，就请返回 -1。

何为山脉数组？如果数组 A 是一个山脉数组的话，那它满足如下条件：

首先，A.length >= 3

其次，在 0 < i < A.length - 1 条件下，存在 i 使得：

  • A[0] < A[1] < ... A[i-1] < A[i]
  • A[i] > A[i+1] > ... > A[A.length - 1]

你将 不能直接访问该山脉数组，必须通过 MountainArray 接口来获取数据：

  • MountainArray.get(k) - 会返回数组中索引为k 的元素（下标从 0 开始）
  • MountainArray.length() - 会返回该数组的长度

注意：

对 MountainArray.get 发起超过 100 次调用的提交将被视为错误答案。此外，任何试图规避判题系统的解决方案都将会导致比赛资格被取消。

思路

一开始想着把左边的有序部分分出来做二分查找，然后把右边的拿去递归。后来发先做出来的结果总是有问题（不是最小下标/调用get超限），打开讨论区看到一个朴素无比的想法：先找到峰值，然后从峰值处把数组分开，这样两边都是有序的。然后先对左边二分查找，再对右边二分查找（左边找不到的情况下）。一开始之所以每这么写是因为怕找峰值比较费时，仔细想想也可以用类似二分查找的方法解决。
